Structuring what the model knows,
not just what you ask it.
context engineering
Role, constraints, examples, output format. Define these once, reuse them across sessions, and the model produces what you need reliably. Leave them out and you're starting from scratch every time.
What lives in a context pack
Five things that change everything.
1
Role
Who the model is in this context. Function, domain, relationship to the work.
2
Goal
What this tool or system is trying to do. Specific, not abstract.
3
Constraints
What counts as out of bounds. Tone rules, regulatory limits, things to avoid.
4
Examples
Real output that represents what good looks like. The model learns from these.
5
Output format
Length, structure, file type, tone register. Specify it and it gets followed.

Build · When something goes wrong
Something's off. Which kind of off?
Context problem
"Does the model know enough about the situation to do this well?"
Fix: Add role, examples, constraints, or domain knowledge to the context.
Instruction problem
"Was I clear enough about what I actually wanted?"
Fix: Rewrite the prompt. Be more specific about format, scope, or outcome.
Scope problem
"Am I asking for too much in one go?"
Fix: Break it into smaller steps. Build in sequence.

Extend · Before you add the API
Where does the key live?
The problem
A key hardcoded in the file
If your key is written into the code, anyone who opens the file can see it. Never push a file with a hardcoded key to GitHub.
What we're doing today
A key pasted at runtime
You paste your key into the tool each time. It stays in browser memory and disappears when you close the tab. Fine for a local tool on your machine.
The production pattern
A key stored in your hosting environment
Vercel, Netlify, or any server. Set the key once in a dashboard. Your code references it as a variable. The key never appears in your files.
The question to always ask
Who can see this, and where does it go?
Local tool, just you? Fine. Shared with a colleague? Remove the key first. Public repo? The key must live somewhere else entirely.

Knowing when to stop
What done looks like.
✓
It does the one thing you said it would do
Not everything you imagined. The thing you scoped.
✓
You can run it, open it, or use it right now
If it requires ten more steps to be useful, it is not done.
✓
A teammate could figure it out without your help
Or future-you, six months from now.
✓
It is better than not having it
That is the bar. Ship it. v2 exists.

"Create a standalone HTML file that explains what we just built.

Include:
1. What it does and why, in plain language
2. A plain-language explanation of the tech stack and architecture
3. A simple diagram showing how the pieces connect
4. An interactive quiz with 3 to 5 questions

Write it like a friendly teacher, not a technical manual."
